iOS_9_10_11_12_适配

发表于 2018-10-12 | 分类于 iOS | 阅读次数:

iOS9适配

1、iOS9引入了新特性APP Transport Security(ATS),所以在Xcode7中发现网络访问失败。并输出错误信息
The resource could not be loaded because the App Transport Security policy requires the use of a secure connection

新特性要求App内访问的网络必须使用 HTTPS 协议。

但是现在公司的项目使用的是 HTTP 协议+HTTPS协议。

最终找到以下解决办法：

在Info.plist中添加 NSAppTransportSecurity 类型 Dictionary 。

在 NSAppTransportSecurity 下添加 NSAllowsArbitraryLoads 类型 Boolean ,值设为 YES

这样将撤回HTTP不安全请求。

2、Bitcode

iOS并不强制使用，但是Xcode7默认会开启Bitcode.

解决方法一：更新第三方库library,使其包含Bitcode

解决方法二：关闭Bitcode

3、URL Scheme

在iOS9中，如果使用URL scheme必须在”Info.plist”中将你要在外部调用的URL scheme列为白名单，否则不能使用。

耗电量优化

发表于 2018-10-09 | 分类于 iOS | 阅读次数:

耗电模块

  1. 屏幕 （亮屏时间）
  2. CPU (复杂运算、无限循环导致CPU过载)
  3. 网络 （网络请求、数据传输、网络切换、大量解析数据等）
  4. GPS
  5. Camera

电量优化最佳实践

  1. 网络相关
  • 发起网络请求时机。业务区分是否需要及时返回（下拉），还是可以延迟执行。
  • 减少网络被激活时间和次数。
  • 避免固定频繁的请求，返回数据相同或者无数据时，推迟下次请求时间。
  • 批处理，合并同一业务的多次请求。
  • 预处理，提前请求某些数据，防止后续频繁请求。
  • 数据处理
  • 网络数据传输前压缩处理
  • 大数据下载使用 GZIP 方式
  • 高效数据格式JSON&Protobuf
  • 慎用轮询方式执行网络请求，iOS采用APNs
  • 减少推送消息次数和频率
  • 处理具体业务前优先判断网络状态。移动网络下，减少数据传输或降低频率（wifi下网络传输 耗电远低于移动网络）；网络不可用下，尽早进入网络异常逻辑，避免不必要的运算
  1. 界面相关
  • 离开页面停止对应耗电活动
  • 进入后台，禁止异常消耗
  1. 定位相关
  • 及时关闭GPS,减少更新频率，根据实际情况切换GPS和网络，不要任何时候都同时使用。
  • 定位要求不高的业务场景，网络定位代替GPS
  • 慎用持续定位，使用一次定位接口即可
  • 慎用被动定位。
  1. 电池状态
  • 处理耗时耗电并且不紧急任务时（如更新包），优先判断电池电量，可以延迟到电量足时执行
  • 监听充电状态，来处理特定业务（更新包，log日志上传、用户数据同步），提高用户体验
  1. 消息广播
  • 灵活控制是否监听广播消息
  1. H5界面

音视频之音频采集Audio Unit

发表于 2018-08-28 | 分类于 iOS | 阅读次数:

音频技术栈

所有 iOS 音频技术都是基于 audio units。此处显示的更高级别的技术，如 Media Player，AV Foundation，OpenAL，AudioToolbox，是对 audio units 的封装，为特定的任务提供专用且简化的 API。

当你需要实现以下需求时，不使用高级 API，直接使用 audio units

  • 低延时音频I/O，例如 VoIP 应用
  • 多路声音的合成回放，例如音乐游戏或合成乐器
  • 使用特定的 audio unit 特征，例如回声消除，混音，均衡器、混响等
  • 需要图状结构来处理音频，可将音频处理模块组装到图状结构中。

Audio Unit工作方式:

应用程序通过一个或者多个回调函数将音频发送到 graph 中的第一个音频单元，并对每一个音频单元进行单独控制。在 I/O unit 的输出(audio processing graph 中最后一个 audio unit)直接连接到输出硬件

Audio Unit 分类

Audio Units 属性

设置属性，可以使用函数 AudioUnitSetProperty

  • kAudioOutputUnitProperty_EnableIO 启用或禁止 I/O，默认输出开启，输入禁止。
  • kAudioUnitProperty_ElementCount 配置元素个数
  • kAudioUnitProperty_MaximumFramesPerSlice 设置 audio unit 的最大帧数
  • kAudioUnitProperty_StreamFormat 指定输入 audio unit 输入输出元素的数据格式
Audio Units的回调函数（AURenderCallback）

  • inRefCon，参数指向回调附加到 audio unit 输入时指定的编程上下文

  • ioActionFlags，参数允许提供提示当没有音频数据处理时，例如当你的应用程序是合成吉他，用户当面没有播放，请执行此操作。当你想要输出静音，可以在回调主体中使用如下语句：*ioActionFlags |= kAudioUnitRenderAction_OutputIsSilence，并且你必须明确地将 ioData 参数指向的缓冲区设置为0。

  • inTimeStamp，回调函数被触发时间，包含一个 AudioTimeStamp 结构体。
  • inBusNumber，参数指示调用回调的音频单元总线
  • inNumberFrames，当前调用的音频采样数
  • ioData，指向音频数据缓存区

Audio Units 结构图

I/O Units 的基本特性

  • Input element 和 Output element 都是 I/O unit 的一部分，可以将它们视为一个独立的个体，单独启动或禁止每一个 Element，默认情况下，Element 1 禁用，Element 0 开启。

  • 音频输入硬件麦克风直接连着 Element 1， Element 1 的 Input scope 对你是不可见的，你首次访问硬件输入的音频数据是位于 Element 1 的 Output scope。

  • 音频输出硬件扬声器直接连着 Element 0，Element 0 的 Output scope 对你是不可见的，数据从 Element 1 的 Output scope 传递到 Element 0 的 Input scope。

每一个 Element都有自己的 input scope 和 output scope，当描述 I/O unit 的时候可能会有困惑，相当于这样描述，你收到收据来自 input element 的 output scope，发送数据到 output element 的 input scope。
I/O unit 是唯一能够在 audio processing graph 中启动和停止音频流的 audio unit。I/O unit 负责在音频单元APP中的音频流。

key 值

key 值必须保证是一个对象级别的唯一常量。
以下三种推荐的 key 值：

  1. 声明 static char kAssociatedObjectKey; ，使用 &kAssociatedObjectKey 作为 key 值;
  2. 声明 static void *kAssociatedObjectKey = &kAssociatedObjectKey; ，使用 kAssociatedObjectKey 作为 key 值；
  3. 用 selector ，使用 getter 方法的名称作为 key 值。

关于block嵌套时，如何正确使用@weakify(self);和@strongify(self);来避免循环引用问题
分以下三种情况
1、局部变量嵌套局部变量 2、成员变量嵌套局部变量 3、成员变量嵌套成员变量

第1种情况，block和nestedBlock作为局部变量，内部使用self不会引起循环引用，所以不必使用@weakify(self);和@strongify(self)

第2种情况，_block作为成员变量，内部使用self会引起循环引用，所以应该使用@weakify(self);和@strongify(self)

第3种情况，_block作为成员变量，内部使用self会引起循环引用，所以应该使用@weakify(self);和@strongify(self)
